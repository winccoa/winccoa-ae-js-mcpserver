  Analyse des aktuellen Browsing-Tools

  Das aktuelle Tool verwendet eine dpConnect-Callback-basierte Implementierung die auf jede Änderung der 
  Browse-Datenpunkte wartet. Dies ist der Hauptgrund für die langsame Performance.

  Aktuelle Probleme:

  1. Zeile 353-365: dpConnect wird verwendet, um auf 7 verschiedene Datenpunkte gleichzeitig zu hören
  2. Zeile 293: Ein einzelner dpGet mit 7 Datenpunkten wird bei jedem Callback ausgeführt
  3. Zeile 283-285: 30 Sekunden Timeout - deutet auf lange Wartezeiten hin
  4. Keine Paginierung: Alle Nodes werden auf einmal zurückgegeben
  5. Keine Filterung: Keine Möglichkeit, nur bestimmte Node-Typen zu browsen
  6. Keine Caching-Mechanismen

● Optimierungsvorschlag für OPC UA Browse Tool

  Fokus auf 2 Hauptoptimierungen:

  Optimierung 1: Batch-Browse / Rekursives Browsing

  Neues Tool: opcua-browse-recursive
  - Browsed mehrere Ebenen in einem Request
  - Parameter maxDepth (z.B. 1-5 Ebenen)
  - Gibt Tree-Struktur zurück statt flacher Liste
  - Vorteil: 1 Request statt N Requests für tiefe Hierarchien
  - Deutlich schneller für große Strukturen

  Beispiel:
  opcua-browse-recursive(connection, "ns=0;i=85", maxDepth=3)
  → Liefert kompletten Tree bis Tiefe 3 in einem Durchgang

  Optimierung 2: In-Memory Browse-Cache

  Implementierung:
  - Einfacher Map-basierter Cache in OpcUaConnection.ts
  - Key: ${connectionName}:${nodeId}:${eventSource}
  - TTL: 5 Minuten (konfigurierbar)
  - Automatisches Cleanup bei Ablauf

  Neue optionale Parameter für opcua-browse:
  - useCache?: boolean (default: true)
  - refreshCache?: boolean (default: false - force refresh)

  Vorteile:
  - Wiederholte Browse-Anfragen instant (0ms statt 2-5s)
  - Besonders effektiv beim "Zurück-Navigieren"
  - Kein zusätzlicher Server-Load
